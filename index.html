從資料表中選取資料。DISTINCT 會過濾掉重複的結果，而預設的 ALL 則包含所有結果。FROM 識別要查詢的資料表（目前僅支援單一資料表 - 尚未支援 JOIN 或衍生資料表）。未在建立時宣告的動態資料行可以在資料表名稱後面的括號中定義，然後在查詢中使用。GROUP BY 根據給定的表達式對結果進行分組。HAVING 在分組後過濾資料列。ORDER BY 根據給定的資料行或表達式對結果進行排序，且僅允許用於聚合查詢或帶有 LIMIT 子句的查詢。LIMIT 限制查詢傳回的資料列數量，如果指定為 null 或小於零則不套用任何限制。LIMIT 子句在 ORDER BY 子句之後執行，以支援 TopN 類型的查詢。可選的 hint (提示) 會覆寫預設的查詢計畫。

範例：

SELECT * FROM TEST;
SELECT a.* FROM TEST;
SELECT DISTINCT NAME FROM TEST;
SELECT ID, COUNT(1) FROM TEST GROUP BY ID;
SELECT NAME, SUM(VAL) FROM TEST GROUP BY NAME HAVING COUNT(1) > 2;
SELECT 'ID' COL, MAX(ID) AS MAX FROM TEST;
SELECT * FROM TEST LIMIT 1000;

UPSERT INTO foo SELECT * FROM bar;

建立一個新的資料表或檢視表。建立資料表時，如果引用的 HBase 資料表和任何資料行族群 (column families) 不存在，則會建立它們（除非使用雙引號引起來才會區分大小寫，否則使用大寫名稱）。未列出的資料行族群不受影響。在建立時，會向任何現有資料列的第一個資料行族群加入一個空的鍵值 (key value)。Upsert 操作也會加入這個空的鍵值。這樣做是為了透過提供一個我們能保證始終存在的鍵值列來提高查詢效能（最小化必須投影的資料量）。或者，如果建立的是檢視表，則 HBase 資料表和資料行族群必須已經存在。不會向現有資料列加入空的鍵值，也不允許進行資料修改 - 檢視表是唯讀的。檢視表的查詢效能將不如真實的資料表。僅對於資料表而言，HBase 資料表和資料行設定選項可以作為鍵/值對傳遞，以便根據需要設定 HBase 資料表。

範例：

CREATE TABLE my_schema.my_table ( id BIGINT not null primary key, date DATE not null)
CREATE TABLE my_table ( id INTEGER not null primary key desc, date DATE not null,
    m.db_utilization DECIMAL, i.db_utilization)
    m.DATA_BLOCK_ENCODING='DIFF'
CREATE TABLE stats.prod_metrics ( host char(50) not null, created_date date not null,
    txn_count bigint CONSTRAINT pk PRIMARY KEY (host, created_date) )
CREATE TABLE IF NOT EXISTS my_table ( id char(10) not null primary key, value integer)
    DATA_BLOCK_ENCODING='NONE',VERSIONS=?,MAX_FILESIZE=2000000 split on (?, ?, ?)

透過新增或移除資料行或更新資料表選項來修改現有資料表。當從資料表中卸除資料行時，該資料行中的資料也會被刪除。主鍵 (PK) 資料行不能被卸除，且只能新增允許 null 值的 PK 資料行。對於檢視表，卸除資料行時資料不受影響。請注意，建立或卸除資料行僅影響後續的查詢和資料修改。在較早時間戳記連線的快照查詢仍會使用資料寫入時有效的舊結構描述。

在資料表或檢視表上建立新的二級索引。隨者資料的變更，索引會自動與資料表保持同步。在查詢時，如果該索引包含查詢中引用的所有資料行，並且能產生最有效率的執行計畫，最佳化工具就會使用該索引。如果資料表具有「只寫一次」且「僅附加」的資料列，那麼資料表可以將 IMMUTABLE_ROWS 屬性設為 true（在 CREATE TABLE 敘述中預先設定，或事後藉由 ALTER TABLE 敘述設定）。這減少了維護索引時的寫入負擔。否則，如果資料表上沒有設定此屬性，那麼當資料更改時，伺服器端將會執行增量的索引維護作業。

範例：

CREATE INDEX my_idx ON sales.opportunity(last_updated_date DESC)
CREATE INDEX my_idx ON log.event(created_date DESC) INCLUDE (name, payload) SALT_BUCKETS=10
CREATE INDEX IF NOT EXISTS my_comp_idx ON server_metrics ( gc_time DESC, created_date DESC )
    DATA_BLOCK_ENCODING='NONE',VERSIONS=?,MAX_FILESIZE=2000000 split on (?, ?, ?)

修改現有索引的狀態。 DISABLE 將導致不再對該索引執行進一步的維護，且它將不再被考慮用於查詢。REBUILD 將完全重建該索引，並在完成後使該索引再次可用於查詢中。UNUSABLE 將導致該索引不再被考慮用於查詢中，然而索引維護仍將繼續執行。USABLE 將導致該索引再次被考慮用於查詢中。請注意，被停用的索引必須重建，且無法直接設為 USABLE。

範例：

ALTER INDEX my_idx ON sales.opportunity DISABLE
ALTER INDEX IF EXISTS my_idx ON server_metrics REBUILD

透過修改相應的 HBase 中繼資料來設定 HBase 資料表或資料行的選項。此選項套用於命名的族群，如果省略且名稱參考了 HColumnDescriptor 屬性，則套用於所有族群。否則，此選項將套用於 HTableDescriptor。

一個內建的選項是 SALT_BUCKETS。此選項會在透明的情況下將一個額外的位元組前置到每個 row key 中，以確保跨所有區域伺服器的寫入負載均勻分佈。這當您的 row key 總是單調遞增，導致在單一區域伺服器上產生熱點時非常有用。此位元組是透過雜湊 row key 並用 SALT_BUCKETS 值求餘數來決定的。該值可以是 1 到 256 之間。如果資料表未定義拆分點 (split points)，那麼它將自動在每個可能的 salt bucket 值處被預先拆分。關於此技術的精彩說明，請參見 http://blog.sematext.com/2012/04/09/hbasewd-avoid-regionserver-hotspotting-despite-writing-records-with-sequential-keys/

另一個內建選項是 IMMUTABLE_ROWS。只有具備不可變資料列的資料表才允許擁有索引。預期不可變資料列只會在完整建立時插入一次，之後永遠不會被更新。一旦實作了漸進式索引維護，這個限制將會被移除。目前的實作在插入資料列時也會一併插入索引列。

範例：

IMMUTABLE_ROWS=true
SALT_BUCKETS=10
DATA_BLOCK_ENCODING='NONE',a.VERSIONS=10
MAX_FILESIZE=2000000000,MEMSTORE_FLUSHSIZE=80000000

Hint

name
 
, ...

用以覆寫預設查詢處理行為的進階功能。支援的 hint 包括：1) SKIP_SCAN 強制對查詢執行略過掃描 (skip scan)，即使原先不會這麼做。如果查詢不包含前端的主要鍵資料行，但包含其他高選擇性的主要鍵資料行時，此選項可提高效能。2) RANGE_SCAN 強制對查詢執行範圍掃描。如果查詢對於非選擇性的前端主要鍵資料行及其他主要鍵資料行一起篩選範圍時，此選項可提高效能。3) NO_INTRA_REGION_PARALLELIZATION 用於防止在單一區域內產生多個執行緒來處理資料。當已知被查詢的整體資料集很小時，此選項很有用。4) NO_INDEX 強制查詢使用原始的資料表。以及 5) INDEX(<table_name> <index_name>...) 用於建議特定查詢該使用哪一個索引。雙引號可用於包圍區分大小寫的 table_name 和 / 或 index_name。

範例：

/*+ SKIP_SCAN */
/*+ RANGE_SCAN */
/*+ NO_INTRA_REGION_PARALLELIZATION */
/*+ NO_INDEX */
/*+ INDEX(employee emp_name_idx emp_start_date_idx) */

備註 (Comments) 可以被用於命令的任何位置且會被資料庫給忽略。行註解的結尾是伴隨著換行字元的。區塊註解不能夠被嵌套建立的，然長度可以有多行之多。
